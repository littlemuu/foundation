# Deque 与迭代器练习提示

## 这次练习的核心

这个简化版 `Deque` 使用一段连续缓冲区模拟环形空间。容器中的逻辑顺序可能是：

```text
逻辑顺序：  3 4 5 6
物理缓冲区：5 6 _ 3 4
```

因此，不能把“第几个元素”和“数组下标”当成同一个概念。`Iterator` 保存的是逻辑位置，解引用时再把它转换成物理下标。

## 需要始终保持的不变量

- `count_ <= capacity_`。
- `front_idx_` 表示第一个元素所在的物理位置。
- `back_idx_` 表示下一次 `push_back()` 写入的位置。
- 空容器中 `count_ == 0`，此时头尾下标应处于一致状态。
- `begin()` 的逻辑位置是 `0`，`end()` 的逻辑位置是 `count_`。
- `end()` 只表示结束位置，不能解引用；但非空容器可以先对它执行 `--`。

## 推荐实现步骤

1. 完成构造和析构，确认缓冲区只申请、释放一次。
2. 完成 `empty()` 和 `size()`。
3. 思考逻辑位置与物理下标的转换规则，完成 `physical_index()`。
4. 完成不涉及扩容的头尾访问和四种 push/pop。
5. 人工画出头尾下标跨过数组末尾时的状态。
6. 完成 `resize()`，重点检查搬移后的元素顺序和头尾下标。
7. 完成迭代器。迭代器移动的是 `pos_`，不是直接移动物理下标。
8. 最后完成 `clear()`、`print()` 和异常检查。

## 扩容时要回答的问题

- 新缓冲区中的第 `0` 个位置应该保存原容器的哪个元素？
- 原来的元素跨越了数组末尾时，怎样仍按逻辑顺序搬移？
- 搬移完成后，新的 `front_idx_` 应该在哪里？
- `back_idx_` 应该指向最后一个元素，还是最后一个元素之后？
- 原缓冲区应该在什么时候释放？

## 迭代器要回答的问题

- 为什么迭代器保存 `Deque<T>*`，而不是只保存一个数组下标？
- 为什么两个 `pos_` 相同、但来自不同容器的迭代器不能相等？
- 为什么前置 `++` 返回引用，而后置 `++` 返回副本？
- `--end()` 为什么可以得到最后一个元素？
- 扩容后旧迭代器是否还应该继续使用？

## 关于模板对象数组

当前框架沿用参考代码中的：

```cpp
new T[capacity_]
```

这会要求 `T` 可以默认构造，并且删除元素时不会立即调用单个元素的析构函数。它适合作为理解环形下标和迭代器的第一阶段练习，但并不是标准库 `std::deque` 的完整内存管理方式。

等核心功能完成后，可以把下面内容作为进阶方向：

- 深拷贝和赋值运算符。
- 移动构造和移动赋值。
- 真正只读的 `const_iterator`。
- 随机访问迭代器的 `+`、`-` 和大小比较。
- 使用原始内存与 placement new 精确控制元素生命周期。

这些内容不属于当前框架的必做部分。

## 实现错误复盘

### 1. `back_idx_` 的含义前后不一致

约定中，`back_idx_` 表示下一次 `push_back()` 写入的位置，而不是最后一个元素的位置。

最初的 `push_back()` 先移动 `back_idx_` 再写入，导致第一次插入就跳过了缓冲区的第 `0` 个位置。修改后应始终遵守下面的顺序：

1. 向 `buffer_[back_idx_]` 写入元素。
2. 将 `back_idx_` 向后移动并处理回绕。
3. 增加 `count_`。

因此，`back()` 不能直接返回 `buffer_[back_idx_]`，因为那里是下一次写入的位置。最后一个元素位于 `back_idx_` 的前一个位置。

`pop_back()` 同样应先把 `back_idx_` 移动到前一个位置，再减少元素数量。

当前状态：`push_back()`、`back()`、只读 `back()` 和 `pop_back()` 均已按照同一个约定修正。

### 2. 无符号下标不能直接执行 `0 - 1`

`back_idx_` 和其他下标的类型是 `std::size_t`，它是无符号整数。下面的代码在 `back_idx_ == 0` 时不会得到 `-1`，而会变成一个很大的正整数：

```cpp
back_idx_ = (back_idx_ - 1) % capacity_;
```

向前移动时需要单独处理下标 `0`：

```cpp
back_idx_ = back_idx_ == 0 ? capacity_ - 1 : back_idx_ - 1;
```

当前状态：`pop_back()` 已单独处理 `back_idx_ == 0`，问题已修正。

### 3. 越界检查应使用元素数量，而不是容量

`capacity_` 表示缓冲区能容纳多少元素，`count_` 才表示当前实际存在多少元素。

最初的 `operator[]` 使用：

```cpp
if (index >= capacity_)
```

这会允许访问尚未存放有效元素的空位置。正确的有效逻辑下标范围是 `[0, count_)`，因此应检查 `index >= count_`。

迭代器的 `operator*()` 和 `operator->()` 也遵守相同规则。尤其是 `end()` 的 `pos_ == count_`，它是结束标记，不能解引用。

当前状态：两个 `operator[]`、迭代器 `operator*()` 和 `operator->()` 均已使用 `count_` 检查有效范围。

### 4. `resize()` 搬移元素时取模对象错误

扩容前，元素的物理位置属于旧缓冲区，因此计算旧下标时应该对旧的 `capacity_` 取模：

```cpp
(front_idx_ + i) % capacity_
```

最初错误地对 `count_` 取模，这会在缓冲区发生回绕或未装满时读到错误位置。

扩容后的状态应满足：

- 元素按照逻辑顺序保存到新缓冲区的 `[0, count_)`。
- `front_idx_ == 0`。
- `back_idx_ == count_`，表示下一个尾插位置。

当前状态：这些下标问题已经修正。

### 5. 替换缓冲区前要释放旧内存

`resize()` 中申请新缓冲区并搬移元素后，必须先释放旧缓冲区，再让 `buffer_` 指向新缓冲区：

```cpp
delete[] buffer_;
buffer_ = new_buffer;
```

如果直接覆盖 `buffer_`，旧缓冲区的地址会丢失，形成内存泄漏。

当前状态：已经修正。

### 6. 循环变量类型应与 `count_` 一致

`count_` 是 `std::size_t`，因此遍历它时使用 `std::size_t i`，可以避免有符号整数与无符号整数比较产生的编译警告。

当前状态：已经修正。

### 7. 零容量需要在构造时处理

如果允许创建 `Deque<int>(0)`，则 `capacity_ * 2` 仍然是 `0`，并且后续取模运算可能发生除零错误。

构造函数应选择一种明确策略：拒绝 `initial_capacity == 0` 并抛出异常，或者把它调整为至少 `1`。

当前状态：构造函数会在 `initial_capacity == 0` 时抛出 `std::out_of_range`，问题已修正。

### 8. 未完成的占位函数会让测试中断

`print()` 最初仍然抛出 `std::logic_error`。即使前面的容器操作全部正确，程序执行到最后也无法通过完整测试。

当前状态：已经实现。

## 最终复查结果

当前待修正清单中的问题均已完成修正。本轮使用 `-std=c++17 -Wall -Wextra -Wconversion -pedantic` 编译，没有产生警告，原有测试全部通过。

另外补充检查了原测试未直接覆盖的边界：

- 构造 `Deque<int>(0)` 时会抛出异常。
- 初始容量为 `1` 时，可以正确触发扩容并保持元素顺序。
- `back_idx_ == 0` 时，`back()` 和 `pop_back()` 可以正确向前回绕。
- 解引用 `end()`，或者通过 `end()->成员` 访问元素，都会抛出 `std::out_of_range`。
- 环形缓冲区发生回绕并扩容后，逻辑顺序保持不变。

在当前简化版练习的目标范围内，暂未发现新的功能错误。

需要注意：迭代器在 `begin()` 前继续执行 `--`，或在 `end()` 后继续执行 `++`，仍属于无效使用。本练习没有在移动运算符中主动检查这些情况，这与当前测试和学习目标并不冲突。
